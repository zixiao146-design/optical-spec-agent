# Local Meep Candidate Hardening v0.6

This document records local/manual hardening evidence for the current v0.6
library-Au physical candidate. It is not production validation and is not part
of ordinary CI.

## Candidate Profile

The candidate preset is `physical-candidate-v0-6`:

| Parameter | Value |

| `boundary_type` | `absorber` |

| `expected_mode` | `research-preview` |
| `physical_interpretation_level` | `physical_candidate` |

This profile is still not production-grade. It is a bounded stability candidate
using fixed runtime, modest resolution, few frequency points, and heuristic
postprocessing.

## Why This Is Not Production Validation

- No convergence study over mesh, runtime, Courant, monitor placement, or flux
  box size has been completed.
- The output spectra are generated by a research-preview script, not a validated
  publication workflow.
- Peak and FWHM extraction are heuristic.
- The profile has only been tested on the current core nanoparticle-on-film case.
- Real Meep execution remains a manual/local gate and is not part of ordinary CI.

## Interpretation

The v0.6 candidate is substantially stronger than the initial pre-study result:

- It has repeatability evidence.
- It tolerates longer fixed runtime.
- It tolerates modest resolution and frequency-count increases.
- It works for both Ex and Ey.
- It produces finite numeric CSVs and valid postprocess JSON across the matrix.

However, it remains a candidate, not production validation. v0.6 can focus on
convergence and physical sanity checks rather than basic execution stability.

## Recommended Next Step

v0.6 should move from stability hardening to physical sanity checks:

- Compare spectra across `fixed_run_time=50/100/200`.
- Increase resolution and frequency points further in a bounded matrix.
- Compare Ex/Ey symmetry expectations.
- Check sensitivity to flux box size and monitor placement.
- Add explicit convergence metrics before any production or publication claims.
